# Техническое задание

**Проект:** ML-система улучшения изображений в браузере   
**Исполнитель:** Ким София Денисовна

---

## 1. Постановка задачи

Разработать веб-систему, которая улучшает качество изображений в реальном
времени, полностью на стороне клиента (в браузере пользователя), без отправки
данных на сервер. Улучшение выполняется по трём параметрам:

- **Яркость** (brightness)
- **Контрастность** (contrast)
- **Цветность / насыщенность** (saturation)

### Принцип решения
> ML-модель **подбирает оптимальные параметры коррекции** (3 коэффициента), а
> **вспомогательный (детерминированный) алгоритм применяет** эти параметры к
> изображению.

Модель **не генерирует** пиксели, а решает задачу **регрессии**: на вход –
уменьшенная копия изображения (128×128), на выход – 3 числа. Это ключевое
архитектурное решение: оно резко снижает размер модели и время инференса.

## 3. Архитектура системы

### 3.1. Порядок работы
1. **Получение** исходного изображения через API
2. **Обработка** с информированием о прогрессе (событие + метод опроса статуса)
3. **Выгрузка** готового изображения

### 3.2. Компоненты

```
┃  UI · главный поток
┃  загрузка файла · превью «до/после» · прогресс · отмена · тонкая настройка
┃  декодирование HEIC → PNG (heic2any требует window, в воркере недоступен)
        │
        │  вызовы API + подписка на событие statusChange
        ▼
┃  ImageEnhancerAPI · фасад, главный поток
┃  createTask · getStatus · cancelTask · getResult · reapplyTask · getPreview
┃  событие statusChange
        │
        │  postMessage (transferable ArrayBuffer для входа)
        ▼
┃  Web Worker · фоновый поток
┃    1. Декодирование JPG/PNG/BMP (HEIC уже пришёл как PNG) + проверка 15 Мпк
┃    2. Thumbnail 128×128 → tensor
┃    3. Инференс TF.js → { brightness, contrast, saturation }
┃    4. Применение коррекции к полному изображению (чанками)
┃    5. Кодирование результата (PNG Blob) + превью оригинала
```

### 3.3. Технологический стек
| Слой | Технология | Обоснование |
|------|-----------|-------------|
| Язык | **TypeScript** | статическая типизация контрактов API между UI и Worker |
| Сборка | **Vite** | tree-shaking, code splitting, ленивые чанки – удержание ≤ 10 МБ |
| ML-рантайм | **TensorFlow.js** | инференс нейросети в браузере без серверной части |
| Бэкенд TF.js | **WebGL → CPU** (фолбэк) | WebGL для скорости, CPU как универсальный фолбэк (в т.ч. Safari-воркер). Модель крошечная, CPU достаточно |
| Фоновый поток | **Web Worker** | асинхронность |
| Обработка пикселей | **OffscreenCanvas** | декодирование и применение коррекции в воркере |
| Декодер HEIC | **heic2any** (ленивый чанк, главный поток) | браузеры нативно HEIC не читают |
| Обучение модели | **Python + TensorFlow/Keras** | обучение CNN, экспорт в TF.js |
| Шрифт/дизайн | **Unbounded** + анимированный градиент, glassmorphism | состояние подсвечивается палитрой (синий/зелёный/красный) |

## 4. Спецификация API модуля

### 4.1. Методы
```ts
interface ImageEnhancerAPI {
  /** Постановка задачи. Возвращает идентификатор задачи. */
  createTask(image: Blob | File | ImageData): Promise<TaskId>;

  /** Текущий статус и прогресс [0..1]. */
  getStatus(taskId: TaskId): Promise<TaskStatusInfo>;

  /** Прерывание задачи. Возвращает признак успеха. */
  cancelTask(taskId: TaskId): Promise<{ success: boolean }>;

  /** Готовое изображение. */
  getResult(taskId: TaskId): Promise<Blob>;
}
```

### 4.1a. Дополнительные методы
```ts
/** Повторное применение коррекции с ручными параметрами (полноразмерно). */
reapplyTask(taskId: TaskId, params: CorrectionParams): Promise<Blob>;
/** Превью оригинала (декодированное воркером) — для отображения и live-превью. */
getPreview(taskId: TaskId): Blob | undefined;
```

### 4.2. Типы
```ts
type TaskId = string;

type TaskStatus =
  | 'queued' | 'decoding' | 'analyzing' | 'applying'
  | 'encoding' | 'done' | 'error' | 'cancelled';

interface TaskStatusInfo {
  taskId: TaskId;
  status: TaskStatus;
  progress: number;          // 0..1
  params?: CorrectionParams; // появляются после стадии analyzing
  error?: string;
}

interface CorrectionParams {
  brightness: number;  // множитель яркости (1.0 — без изменений)
  contrast: number;    // множитель контраста
  saturation: number;  // множитель насыщенности
}
```

### 4.3. События
```ts
/** Возникает при изменении статуса ИЛИ прогресса задачи. */
api.on('statusChange', (info: TaskStatusInfo) => void);
```

### 4.4. Отказоустойчивость
- Любая ошибка декодирования/инференса → `status: 'error'` + поле `error`, без падения приложения.
- Превышение N3 (>15 Мпк) → понятная ошибка до начала обработки.
- Фолбэк бэкендов TF.js: **WebGL → CPU**; при сбое загрузки модели – повторная попытка, плюс мягкий фолбэк на единичные коэффициенты (изображение не портится).
- HEIC без поддержки декодера → информативное сообщение.
- `cancelTask` корректно прерывает на любой стадии (флаг отмены проверяется между этапами и внутри попиксельного цикла чанками).

## 5. ML-модель

### 5.1. Тип задачи
**Регрессия:** изображение → 3 вещественных числа (коэффициенты коррекции).

### 5.2. Архитектура
- Вход: thumbnail **128×128×3**, нормализованный в [0,1].
- Тело: компактная свёрточная сеть (4 блока Conv+BN+ReLU+MaxPool, 16→32→48→64).
- Голова: GlobalAveragePooling → Dense(32) → **3 выхода** (softplus, положительные множители).
- Размер: **~49 тыс. параметров → 112 КБ** после квантования (с огромным запасом под 10 МБ).

### 5.3. Данные и цель обучения (self-supervised, авто-улучшение)
Итоговый подход – обучение модели приводить изображение к **эталонной
тональной/цветовой статистике** (а не «отменять случайное искажение»):

1. Берём набор реальных фотографий (открытые фото, скачиваются скриптом).
2. Создаём разнообразие **входов**: с вероятностью 0.8 случайно искажаем кадр
   (яркость/контраст/насыщенность), остальное – как есть.
3. Для каждого входа **аналитически вычисляем "идеальные" факторы коррекции**:
   - тон – мягкие авто-уровни по перцентилям гистограммы, разложенные в (яркость,
     контраст) через аффинное преобразование `out = A·v + B`, где `A = b·c`;
   - насыщенность – подтяжка средней HSV-насыщенности к целевой.
4. Модель учится предсказывать эти факторы (loss MSE).

Преимущество: нормальные фото получают аккуратное улучшение (а не игнорируются),
плохие – сильное; целевые значения вычисляются точно, разметка не нужна.

### 5.4. Метрики
- Loss: MSE по 3 коэффициентам; валидационный MAE.
- Прокси-качество: **PSNR / SSIM** между восстановленным и оригиналом на пуле
  эталонных изображений (см. §8).

### 5.5. Экспорт
Keras (.h5) → `tensorflowjs_converter` → `model.json` + бинарные шарды весов.
Квантование весов **uint16** для уменьшения размера.
Особенность: конвертер разбивает аргументы по пробелам, поэтому запускается из
каталога `training/` с относительными путями (путь проекта содержит пробел).

## 6. Вспомогательный алгоритм применения коррекции

Применяется к полному изображению (OffscreenCanvas / ImageData):

- **Контраст вокруг средней яркости изображения** (а не вокруг 128) – чтобы
  усиление контраста не гасило осветление тёмных кадров. Тон – аффинное:
  `out = brightness · ((v − mean) · contrast + mean)`, где `mean` – средняя luma.
- **Насыщенность:** luma `Y = 0.299R + 0.587G + 0.114B`, затем `out = Y + (v − Y) · s`.
- Клиппинг [0,255]. Яркость и контраст считаются через LUT (256 значений) для скорости.

Обработка идёт чанками (по ~500к пикселей) с отчётом о прогрессе и проверкой
флага отмены – критично для 15 Мпк.

## 7. Производительность (как укладываемся в лимиты)

- Инференс – по уменьшенному thumbnail (128²), а не по полному изображению.
- Применение коррекции – LUT + простые операции; чанками, без блокировки воркера.
- Web Worker + transferable (`ArrayBuffer`) → UI свободен.
- Бандл: ленивые чанки (TF.js ~1.9 МБ и heic2any ~1.35 МБ грузятся по требованию),
  квантованная модель 112 КБ → стартовая загрузка ~28 КБ, всё решение 3.3 МБ.
- Память: освобождение тензоров (`tf.tidy`), очистка данных прошлых задач;
  профилирование – heap стабилен на повторных прогонах 15 Мпк (без утечек).

## 8. Пул эталонных изображений (для оценки качества)

Набор тестовых изображений, покрывающий кейсы:
- **Искажения:** тёмные / пересвеченные / низкоконтрастные / блёклые / смешанные
- **Форматы:** JPG, PNG, HEIC, BMP
- **Разрешения:** вплоть до 15 Мпк
- **Граничные:** очень маленькие, ч/б, уже хорошие (модель не должна «портить»)

Реализация: `eval/evaluate.py` скачивает свежие фото (не из обучения), формирует
кейсы, прогоняет полный пайплайн (модель → коррекция теми же формулами, что в
рантайме), считает PSNR/SSIM и время, сохраняет наглядные примеры и отчёт
`eval/report.md`. Формат-специфичное декодирование (HEIC/BMP) дополнительно
проверяется в браузере.
